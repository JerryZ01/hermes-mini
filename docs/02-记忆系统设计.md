## 2.1 为什么记忆是 Agent 的分水岭

用一个场景说明：

> **无记忆 Agent**
> 第一次对话：用户说"我叫杰哥"
> 第二次对话：用户说"你好"，Agent 回复"你好！有什么可以帮你？"
> 问题：Agent 完全不记得用户的名字，对话是孤立的

> **有记忆 Agent**
> 第一次对话：用户说"我叫杰哥"，记忆系统记录"用户名叫杰哥"
> 第二次对话：用户说"你好"，Agent 检索记忆，回复"你好杰哥！"
> 优势：越来越懂用户，越用越顺手

**记忆系统解决的三个问题：**

1. **跨会话记忆**：关掉再打开，Agent 认识你
2. **上下文注入**：当前对话中自动带入相关信息
3. **用户建模**：不只是记住"说了什么"，而是记住"你是什么样的人"

## 2.2 三层记忆架构

```
┌─────────────────────────────────────┐
│  Honcho 用户画像层（最顶层）          │ ← 主动注入用户特征
├─────────────────────────────────────┤
│  Long-term Memory 长期记忆层         │ ← SQLite FTS5，跨会话持久化
├─────────────────────────────────────┤
│  Working Memory 工作记忆层（最底层）   │ ← 当前会话的消息历史
└─────────────────────────────────────┘
```

**各层职责：**

| 层级 | 内容 | 生命周期 | 容量 |
|------|------|----------|------|
| Working Memory | 当前会话消息历史 | 当前会话 | 受 token 上限限制 |
| Long-term Memory | 关键对话摘要 | 永久 | 无限，但需检索 |
| Honcho Profile | 用户特征标签 | 永久 | 有限（几十个标签） |

## 2.3 Working Memory（工作记忆）

**原理：**

工作记忆就是当前会话的消息列表。每次用户发消息，Agent 把消息 append 进列表，然后发给 LLM。

```python
# 每次对话开始，加载历史消息
self.messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好杰哥！"},
]
```

**Token 上限问题：**

LLM 有 context 窗口限制（如 GLM-5 是 128K，Qwen 有 32K / 128K 不同版本）。当消息历史超过上限时，需要**上下文压缩**。

**上下文压缩策略（简化版）：**

```python
def compress(self, llm_summarize_fn, keep_recent: int = 4):
    # 1. 让 LLM 把历史对话提炼成摘要
    history_text = "\n".join(
        f"[{m['role']}] {m['content'][:200]}"
        for m in self.messages
    )
    prompt = f"请提炼以下对话的核心内容：{history_text}"
    summary = llm_summarize_fn(prompt)

    # 2. 保留 system + 摘要 + 最近 N 轮
    self.messages = [
        system_msg,
        {"role": "system", "content": f"【历史摘要】{summary}"},
    ] + self.messages[-keep_recent * 2:]
```

真实系统（Claude Code、OpenAI）会用更复杂的压缩算法，如按 token 计数 + 语义压缩。Hermes-mini 用简化版便于教学理解。

## 2.4 Long-term Memory（长期记忆）

**为什么需要 FTS5？**

长期记忆需要快速检索。假设 Agent 和用户聊了 100 次对话，5000 条消息，要找"上次聊到的 Python 协程"不能线性扫描。

方案：**SQLite FTS5 全文索引**。

```sql
-- 建表
CREATE VIRTUAL TABLE memories USING fts5(
    keywords,   -- 关键词（用于检索）
    content,    -- 记忆内容
    session_id, -- 所属会话
    timestamp   -- 时间戳
);
```

FTS5 是 SQLite 内置的全文搜索引擎，支持：
- 关键词匹配：`SELECT ... WHERE memories MATCH 'Python 协程'`
- 排序：按相关性 rank 排序
- 轻量：无须安装额外依赖

**检索流程：**

```
用户：上次你说的 Python 协程例子在哪里？
       ↓
检索关键词："Python" "协程" "例子"
       ↓
FTS5 查询 → 找到相关记忆
       ↓
LLM 总结 → "上次讨论了 asyncio.gather 的用法，示例代码在..."
       ↓
注入上下文 → Agent 引用这个记忆回复用户
```

**为什么不直接用向量数据库（Embedding）？**

向量数据库适合语义相似性检索（如"找和'音乐'相关的记忆"），但：
1. 需要额外的服务（Milvus / Pinecone / Qdrant）
2. Embedding 模型有成本
3. 对于关键词明确的任务，FTS5 更简单且更精确

Hermes-mini 选择 FTS5，因为它是 SQLite 内置的零成本方案。

## 2.5 Honcho 用户画像

这是 Hermes 最有特色的设计之一。

**Honcho 解决什么问题？**

Long-term Memory 记住了"做过的事"，但没有记住"人"。Honcho 提炼的是用户的**特征模式**：

- 偏好简洁还是详细的回答？
- 习惯用中文还是英文？
- 对什么话题特别感兴趣？

这些特征帮助 Agent **主动调整自己的风格**，而不需要用户每次都重复说"你说话要简洁一点"。

**实现原理（简化版）：**

```python
def update(self, user_input: str, assistant_output: str):
    # 语言检测
    has_chinese = any('\u4e00' <= c <= '\u9fff' for c in user_input)
    has_english = any(c.isascii() and c.isalpha() for c in user_input)
    if has_chinese and not has_english:
        self.profile["语言偏好"] = "倾向使用中文"
        self._save()

    # 风格检测（通过输入长度推断）
    if len(user_input) < 30:
        self.profile["回答风格"] = "偏好简洁直接"
```

真实的 Honcho（[plastic-labs/honcho](https://github.com/plastic-labs/honcho)）使用 LLM 来提炼用户特征，
每次对话后调用 LLM 分析并更新画像，精度更高但成本也更高。

## 2.6 三层协作：一次完整对话的记忆旅程

```
用户："帮我写一个异步爬虫"

Step 1: Honcho 查询
  → 发现用户"偏好简洁回答，不喜欢废话"
  → 注入到 system prompt

Step 2: Long-term Memory 查询
  → 检索"异步"、"爬虫"、"代码"
  → 找到上次写的 aiohttp 示例
  → 注入上下文："上次你讨论了 asyncio.gather..."

Step 3: Working Memory 管理
  → 加入用户消息
  → 调用 Agent Loop
  → 必要时压缩历史

Step 4: 对话结束后
  → Honcho 更新画像（如发现用户很懂技术）
  → Long-term Memory 存储关键内容
  → Working Memory 随会话结束清空
```

## 2.7 记忆系统的工程挑战

**挑战 1：什么时候存储记忆？**

不能什么都记（太贵），也不能什么都不记（没意义）。

策略：
- 包含关键决策或结论的对话：✓ 存储
- 日常寒暄：✗ 不存储
- 触发词（"以后都"、"记住了"）：✓ 存储

**挑战 2：记忆检索的精度**

简单关键词匹配不够，但引入 Embedding 成本高。

实用折中：FTS5 关键词 + LLM 总结重排序（先粗筛再精排）

**挑战 3：记忆的时效性**

用户偏好可能变化（如换了工作领域），记忆需要定期清理或降权。
