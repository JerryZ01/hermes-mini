## 3.1 工具是 Agent 的"手脚"

没有工具的 Agent 只能"说话"，不能改变外部世界。

有工具之后，Agent 可以：
- 读取本地文件，分析代码
- 执行 Shell 命令，运行测试
- 搜索网页，获取最新信息
- 读写数据库，操作 API
- 发送消息，发邮件

这就是 **"Chatbot" 和 "Agent" 的本质区别**：
- Chatbot：影响"数字世界"（文字）
- Agent：影响"物理世界"（文件、服务器、网络）

---

## 3.2 工具调用的完整链路

```
用户：帮我查一下上海的天气

Agent Loop 推理：
  "我需要调用天气查询工具，参数是 city=上海"
       ↓
【工具调用】LLM 输出 JSON
  {"name": "weather", "args": {"city": "上海"}}
       ↓
【工具执行】hermes-agent 调用天气 API
  API 返回："上海，多云，22度"
       ↓
【结果注入】把结果注入上下文
  messages.append({
    "role": "user",
    "content": "[工具结果] weather: 上海，多云，22度\n请继续"
  })
       ↓
Agent Loop 继续推理：
  "有了天气数据，我可以回答用户的问题了"
       ↓
最终回复：今天上海多云22度，适合出门
```

**关键理解**：工具调用不是"函数调用"——它是 LLM 的一种输出格式，通过 JSON 字符串约定，代码解析后执行。

## 3.3 工具注册表设计

```python
TOOL_REGISTRY: dict[str, ToolDefinition] = {}

@register_tool("天气查询", "查询指定城市的实时天气")
def weather(city: str) -> str:
    return f"{city}今天晴，25度"
```

`TOOL_REGISTRY` 是全局字典，key 是工具名，value 是 `ToolDefinition` 对象。

**为什么用装饰器而不是直接注册？**

装饰器的好处：工具定义和注册写在一起，新增工具时不需要修改任何其他代码。

```python
# 新增一个工具，只需这样：
@register_tool("计算器", "执行数学计算")
def calculator(expr: str) -> str:
    ...

# Agent Loop 不需要改动，registry 里已经自动有了
```

这叫**开放封闭原则**（Open-Closed Principle）：对扩展开放，对修改封闭。

## 3.4 工具的元数据：为什么需要 Schema？

Agent 调用工具时，LLM 需要知道：
- 有哪些工具可以调用？
- 每个工具接受什么参数？
- 参数的类型是什么？

这些信息通过工具的 JSON Schema 传递给 LLM：

```json
{
  "name": "天气查询",
  "description": "查询指定城市的天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称"
      }
    },
    "required": ["city"]
  }
}
```

Hermes-mini 从函数签名自动生成 Schema：

```python
# 从 inspect.signature 自动提取参数信息
sig = inspect.signature(weather)
params = {
    p.name: {
        "type": "string",
        "default": p.default if p.default != inspect.Parameter.empty else None,
    }
    for p in sig.parameters.values()
}
```

## 3.5 内置工具详解

### 计算器

```python
@register_tool("计算器", "安全执行数学表达式")
def calculator(expr: str) -> str:
    # 安全检查：只允许数字和运算符
    if not re.match(r"^[\d\s\+\-\*\/\.\(\)]+$", expr):
        return "[错误] 包含非法字符"
    result = eval(expr, {"__builtins__": {}})
    return str(result)
```

**安全要点**：这里用的是"白名单字符检查 + 禁用 `__builtins__`"的沙箱方案。
真实系统应该用 `ast.literal_eval` 或专门的数学表达式解析器。

### 读文件

```python
@register_tool("读文件", "读取文件内容")
def read_file(path: str) -> str:
    content = Path(path).read_text(encoding="utf-8")
    if len(content) > 50_000:
        return content[:50_000] + "\n[文件过长，已截断]"
    return content
```

这是 Hermes Agent 里最重要的工具之一。Agent 有了文件读写能力，才能：
- 读取代码库，自主分析
- 读取日志，定位 bug
- 写入文件，生成代码

### 搜索

```python
@register_tool("搜索", "在 Google 上搜索关键词")
def web_search(query: str) -> str:
    resp = requests.get(
        "https://api.duckduckgo.com/",
        params={"q": query, "format": "json"},
        timeout=10,
    )
    results = resp.json().get("RelatedTopics", [])
    return "\n".join(r.get("Text") for r in results[:5])
```

这里用的是 DuckDuckGo 免 API Key 接口，适合演示。生产环境建议用 Serper API 或 Google Custom Search API。

## 3.6 工具结果注入的格式设计

工具结果注入到上下文时，格式很关键：

```python
self.messages.append({
    "role": "user",  # 注意：是 user 而不是 tool
    "content": (
        f"[工具调用结果]\n"
        f"工具: {tool_name}\n"
        f"参数: {json.dumps(tool_args)}\n"
        f"结果: {tool_result}\n\n"
        "请基于工具结果继续推理，或输出最终回答。"
    ),
})
```

**为什么用 `role: "user"` 而不是单独的 `"tool"` 角色？**

这是 OpenAI Chat Completions API 的格式设计：
- `role: "tool"` 是函数调用时的特殊格式（`tool_calls` 字段）
- 工具结果作为 `tool` 角色的消息，LLM 才能正确关联到之前的调用

Hermes-mini 用简化的 JSON 字符串方案，所以用 `user` 角色注入结果——效果相同但更简单。

## 3.7 工具调用的常见问题与解决

### 问题 1：LLM 乱传参数

LLM 可能传入不存在的工具名或不匹配的参数类型。

解决：注册时检查参数合法性，执行时 try-catch 捕获异常：

```python
def call_tool(name: str, args: dict) -> str:
    if name not in TOOL_REGISTRY:
        return f"[错误] 未知工具: {name}，可用: {list(TOOL_REGISTRY.keys())}"
    try:
        return TOOL_REGISTRY[name].invoke(args)
    except TypeError as e:
        # 参数缺失或类型错误，返回友好提示
        return f"[错误] 参数不匹配: {e}"
    except Exception as e:
        # 其他错误
        return f"[错误] {name} 执行失败: {e}"
```

### 问题 2：LLM 不调用工具（遇到计算任务硬撑着回答）

解决：在 system prompt 中明确指令："如果需要执行计算，请调用计算器，不要自己硬算"。

### 问题 3：工具执行超时

解决：给每个工具调用加 timeout，并返回超时错误信息，不让 Agent 陷入等待。
